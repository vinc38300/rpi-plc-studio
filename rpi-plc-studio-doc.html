🎯 Aperçu général

RPi-PLC Studio est un environnement de développement complet pour créer, tester et déployer des programmes d'automate sur un Raspberry Pi 3. Il combine un éditeur graphique FBD (Function Block Diagram) sur PC, un moteur PLC autonome sur RPi, et une interface SCADA accessible depuis n'importe quel navigateur du réseau local.

Le système est conçu pour une application réelle de régulation thermique avec :

  • 12 sondes NTC40 connectées via 3 convertisseurs ADS1115 (pont diviseur 10 kΩ / 3,3 V)
  • 10 sorties relais K1 à K10 pour contacteurs, vannes, circulateurs
  • 8 entrées TOR pour boutons-poussoirs, pressostatats, contacts
  • 60+ blocs FBD disponibles dont CArithm (code C embarqué), PID, LOCALTIME, RUNTIMCNT, CONTACTOR, VALVE3V…
  • SCADA 11 onglets + éditeur de synoptique personnalisable
  • Bot Telegram — alertes automatiques et commandes à distance
  • Calibration par sonde, recettes de consignes, sauvegardes versionnées
  • Application mobile PWA installable, fonctionnement hors-ligne

Flux de travail typique

1
Dessiner le programme sur PC Créer le schéma FBD dans l'éditeur graphique, connecter les blocs, configurer les paramètres.
2
Tester en simulation Le moteur PLC tourne en local avec des valeurs simulées. Les curseurs permettent de forcer les entrées analogiques.
3
Déployer sur le RPi Un clic sur "Déployer" envoie le programme par SSH. Le RPi l'exécute de façon autonome, même sans réseau.
4
Superviser via SCADA Depuis un téléphone, tablette ou PC, accéder à l'interface web pour surveiller les températures, modifier les consignes, consulter l'historique.
5
Personnaliser le synoptique Dessiner son propre écran de supervision avec les widgets adaptés à l'installation (thermomètres, jauges, boutons, tuyauteries…).
6
Recevoir les alertes Le bot Telegram notifie immédiatement en cas de dépassement de seuil et permet de piloter l'automate par commandes texte.

⚙ Installation

Prérequis PC (Kubuntu / Ubuntu)

# Dépendances système
sudo apt install python3-pyqt5 python3-pyqt5.qtwebengine \
                 python3-pyqt5.qtwebchannel python3-paramiko

# Lancer l'application
cd rpi-plc-studio
python3 main.py

Installation sur le Raspberry Pi

# Copier le dossier rpi_server/ sur le RPi, puis :
cd ~/rpi_server
chmod +x setup_autonomy.sh
./setup_autonomy.sh

# Dépendances Python
pip3 install flask flask-socketio smbus2 --break-system-packages

# Activer le démarrage automatique
sudo cp rpi-plc.service /etc/systemd/system/
sudo systemctl enable rpi-plc
sudo systemctl start rpi-plc

Activer I²C sur le Raspberry Pi

sudo raspi-config
# Interface Options → I2C → Enable

# Vérifier la détection des 3 ADS1115
sudo i2cdetect -y 1
# Doit afficher : 48, 49, 4a
✅

Vérification rapide : Si i2cdetect affiche bien 0x48, 0x49 et 0x4A, les 3 ADS1115 sont correctement câblés et l'I²C est actif.

📌 Configurateur GPIO

Le configurateur GPIO (Raspberry Pi → Configurer les GPIO…, raccourci Ctrl+G) permet de définir visuellement le rôle de chaque broche BCM du Raspberry Pi, sans éditer config.json à la main. La configuration est sauvegardée dans le projet .plcproj et automatiquement poussée vers le canvas FBD et le config.json du serveur lors du déploiement.

Principe de fonctionnement

Chaque ligne du configurateur représente une broche GPIO et comporte les champs suivants :

ChampValeursDescription
Pin BCM4–27 (sauf 2, 3)Numéro de broche en numérotation BCM. Les GPIO 2 (SDA) et 3 (SCL) sont bloqués car réservés à l'I²C.
NomTexte libreLibellé affiché dans le SCADA et les blocs FBD (ex : Sortie K1, Entrée TOR 3).
ModeEntrée / SortieDirection de la broche.
Pullup / down / offRésistance de tirage interne (pour les entrées uniquement).
Actif bas☑ / ☐Si coché, la logique est inversée : GPIO à 0 V = sortie active (typique des relais opto-couplés).

Broches réservées et avertissements

GPIO BCMStatutRaison
GPIO 2 (SDA)🔒 BloquéI²C SDA — bus des cartes ADS1115. Non configurable.
GPIO 3 (SCL)🔒 BloquéI²C SCL — bus des cartes ADS1115. Non configurable.
GPIO 14 (TX)⚠ AvertissementUART TX — peut interférer avec la console série. Utilisable si la console série est désactivée.
GPIO 15 (RX)⚠ AvertissementUART RX — même remarque.
GPIO 18 (PCM_CLK)⚠ AvertissementSignal audio PCM. Si utilisé en sortie, setup_autonomy.sh désactive automatiquement dtparam=audio dans config.txt.

Workflow typique

1
Ouvrir le configurateur via Raspberry Pi → Configurer les GPIO… (Ctrl+G).
2
Ajouter une ligne par broche à utiliser. Choisir le pin BCM, le nom, le mode et les options pull/actif-bas.
3
Valider — la config est sauvegardée dans plc.gpio_config du projet .plcproj.
4
Déployer via 🚀 Déployer — le config.json du serveur RPi est mis à jour avec la section "gpio" correspondante.
💡

La configuration GPIO est immédiatement reflétée dans les blocs INPUT et OUTPUT du canvas FBD : les listes déroulantes de broches utilisent les noms définis dans le configurateur.

ℹ

La configuration GPIO est stockée dans deux endroits : dans le projet .plcproj (champ plc.gpio_config) pour l'édition PC, et dans rpi_server/config.json (section "gpio") pour le serveur RPi. Le déploiement synchronise les deux.

⊞ Blocs Groupes — Encapsulation et bibliothèque

Les blocs Groupes permettent d'encapsuler un sous-programme FBD à l'intérieur d'un bloc unique réutilisable. Un groupe se comporte comme une boîte noire avec des ports d'entrée/sortie nommés — on câble l'extérieur sans voir l'intérieur. Les groupes peuvent être imbriqués (groupe dans un groupe) et sauvegardés dans une bibliothèque personnelle persistante.

GROUP Bloc groupe
Contient un sous-programme FBD complet. Les ports sont déterminés par les blocs GROUP_IN et GROUP_OUT placés à l'intérieur. Double-clic pour entrer, Échap ou bouton ← Sortir pour revenir au niveau parent.
INx (variable)Qx (variable)
GROUP_IN Port d'entrée du groupe
Placé à l'intérieur d'un groupe. Son port de sortie SIG transmet la valeur reçue de l'extérieur vers les blocs internes. Le libellé (renommable dans les propriétés) devient le nom du port visible sur le bloc GROUP parent.
SIG
GROUP_OUT Port de sortie du groupe
Placé à l'intérieur d'un groupe. Son entrée IN reçoit le signal calculé à l'intérieur, qui est exposé comme port de sortie sur le bloc GROUP parent. Le libellé devient le nom du port externe.
IN

Créer un groupe — méthode 1 : grouper des blocs existants

1
Sélectionner plusieurs blocs en maintenant Ctrl et en cliquant sur chacun (ou par un rectangle de sélection).
2
Cliquer ⊞ Grouper dans la barre de navigation, ou appuyer sur Ctrl+G.
3
Nommer le groupe dans la boîte de dialogue.
4
Les blocs sélectionnés disparaissent de la page et sont encapsulés dans le nouveau bloc GROUP. Les fils coupés deviennent automatiquement des ports GROUP_IN (signaux entrants) et GROUP_OUT (signaux sortants).

Créer un groupe — méthode 2 : bloc GROUP vide

Glisser GROUP depuis la palette (catégorie Groupes), nommer le groupe, puis double-cliquer pour entrer dans le groupe et y placer des blocs. Ajouter des blocs GROUP_IN et GROUP_OUT pour définir les ports.

Ports — nommage et ordre

  • Le nom d'un port correspond au champ label du bloc GROUP_IN ou GROUP_OUT à l'intérieur. Il se modifie dans le panneau propriétés quand le bloc interne est sélectionné.
  • L'ordre vertical des ports sur le bloc GROUP externe suit la position verticale (y) des blocs GROUP_IN / GROUP_OUT à l'intérieur — les placer de haut en bas dans l'ordre souhaité.
  • Ajouter ou supprimer un GROUP_IN / GROUP_OUT à l'intérieur met à jour les ports du bloc GROUP externe dès la sortie du groupe.

Dégrouper

Sélectionner un bloc GROUP → appuyer sur Ctrl+Maj+G. Les blocs internes sont replacés sur la page courante, les fils internes sont restaurés, et les fils qui traversaient la frontière du groupe sont reconnectés directement aux blocs réels. Les blocs GROUP_IN et GROUP_OUT intermédiaires sont supprimés.

💡

La reconnexion des fils externes est automatique : un fil qui arrivait sur le port IN1 du groupe est rebranché directement sur l'entrée du bloc interne qui recevait ce signal. Idem pour les fils sortants.

Groupes imbriqués

Un groupe peut contenir d'autres groupes. Il n'y a pas de limite de profondeur. Le fil d'ariane affiche l'ensemble de la hiérarchie : Page1 › RegulChauffage › ZoneRDC › PID. Sortir d'un niveau profond sauvegarde automatiquement chaque niveau intermédiaire. L'aplatissement récursif gère jusqu'à 32 niveaux d'imbrication.

Comportement à l'exécution

Avant chaque exécution (simulation PC ou déploiement RPi), les blocs GROUP sont aplatis de façon transparente : la hiérarchie est développée récursivement et seuls les blocs feuilles réels sont transmis au moteur PLC. Ce processus se fait en trois endroits :

ContexteOùFonction
Conversion FBD → programme (PC)block_editor.py_flatten_page() appelée dans _fbd_to_program()
Chargement au boot RPiserver.pyflatten_blocks() après load_program_file()
Déploiement via API (POST /api/program)server.pyflatten_blocks() avant engine.load_program()
Restauration sauvegardeserver.pyflatten_blocks() avant engine.load_program()

Les blocs GROUP_IN et GROUP_OUT n'ont aucune existence dans le moteur — ils servent uniquement à l'interface graphique pour nommer les ports. Ils ne créent pas de délai ni de copie de valeur supplémentaire.

💻 Bloc CArithm — Code C embarqué

Le bloc CArithm est le bloc le plus puissant du système. Il permet d'intégrer du code algorithmique complexe directement dans le schéma FBD, exactement comme le bloc CArithm de Proview. C'est la solution pour toutes les logiques qui ne peuvent pas s'exprimer simplement avec des blocs standards.

Principe de fonctionnement

Le code est écrit en syntaxe type C et interprété à chaque cycle du PLC. Il dispose de variables prédéfinies :

VariablesTypeRôleNombre
A1..A8floatEntrées analogiques (registres RF ou sondes)0 à 8
d1..d7boolEntrées booléennes (bits M ou GPIO)0 à 7
I1, I2intEntrées entières (heure, jour…)0 à 2
OA1..OA8floatSorties analogiques (vers registres RF)0 à 8
od1..od8bool (0/1)Sorties booléennes (vers bits M ou GPIO)0 à 8
OI1intSortie entière0 ou 1

Configuration dans le panneau propriétés

  1. Définir le nombre d'entrées (n_a, n_d, n_i) → les ports apparaissent sur le bloc
  2. Définir le nombre de sorties (n_oa, n_od, n_oi)
  3. Écrire le code dans l'éditeur — la touche Tab insère une indentation
  4. Le code est sauvegardé en temps réel dans le projet

Exemple — Loi d'eau (tiré de Proview CArit75)

// A1 = Correction solaire    A2 = Temp extérieur
// A6 = Consigne loi d'eau    d1 = V3V en position solaire
// OA1 = Consigne chaudière   OA2 = Consigne solaire

// Loi d'eau chaudière
OA1 = (0.6 * (A6 - A2) + A6);

// Consigne solaire : avec correction si V3V OK
if (d1) OA2 = OA1 + A1;
else OA2 = OA1;

Exemple — Programmation horaire (tiré de CArit148)

// A1 = heure de début    A2 = heure de fin
// I1 = heure courante (depuis LOCALTIME.HOUR)
// od2 = signal "début"    od4 = signal "fin" (reset SR)

if (A1 == I1) od2 = 1;
else od2 = 0;

if (A2 == I1) od4 = 1;
else od4 = 0;

Exemple — Gestion ECS ballon (tiré de CArit81)

// A1=Temp solaire  A2=Temp sonde haut  A5=Consigne ballon
// A6=Hystérésis haut

OA1 = (A5 + A6);        // Consigne haute corrigée

// Circulateur solaire si différence de temp suffisante
if ((A2 + 3) < A1) od1 = 1;
else od1 = 0;

// Demande de chauffe si ballon insuffisant
if (A5 < OA1) od6 = 1;
else od6 = 0;
⚠

Le code CArithm est converti en Python et exécuté avec exec(). Les boucles (for, while) et les appels de fonctions externes ne sont pas supportés. Seuls if/else et les opérations arithmétiques sont permis.

Principe de mesure NTC40 par pont diviseur

Chaque sonde NTC40 (thermistance NTC, R₀ = 40 kΩ à 25°C) est montée en pont diviseur avec une résistance de référence de 10 kΩ :

+3,3V ──── R_ref (10 kΩ) ────┬──── NTC40 (Rx) ──── GND │ ADS1115 (IN+) mesure Vm

La conversion tension → température se fait par la formule :

# Résistance de la sonde depuis la tension mesurée
Rx = Vm × R_ref / (Vcc - Vm)

# Température via équation de Steinhart-Hart (modèle β) — NTC40
# R0 = 40 000 Ω (résistance NTC à 25°C), β = 3950 K, T0 = 298.15 K
import math
T_kelvin = 1 / (1/298.15 + (1/3950) * math.log(Rx / 40000))
T_celsius = T_kelvin - 273.15
# NB : pour PT100 → T = (Rx - 100) / (100 × 0.00385)
#      pour NTC10K → ntc_r0 = 10000 au lieu de 40000

🔌 Bibliothèque GPIO — gpiod v2 / lgpio

RPi-PLC Studio utilise la bibliothèque GPIO moderne gpiod v2 (API officielle Linux pour Raspberry Pi OS Bookworm 64-bit), avec un mécanisme de fallback progressif vers lgpio, puis un mode simulation si aucune bibliothèque n'est disponible.

Priorité de sélection au démarrage

PrioritéBibliothèqueInterfaceCas d'usage
1gpiod v2 (python3-gpiod ≥ 2.0)/dev/gpiochip0Raspberry Pi OS Bookworm 64-bit — recommandé
2lgpio/dev/gpiochip0Bullseye, systèmes sans gpiod v2
3Mode simulationVariables mémoirePC de développement, tests sans matériel
ℹ

RPi.GPIO n'est plus utilisé depuis Raspberry Pi OS Bookworm 64-bit (le module est retiré des dépôts). gpiod v2 est l'API officielle qui le remplace, avec une gestion par caractéristique (chip, line, request) plutôt que par numéro de pin global.

# Vérifier lgpio

# Vérifier /dev/gpiochip0

# Dans les logs du serveur :
journalctl -u rpi-plc -n 30 | grep -i gpio

Conflit de ressource GPIO

gpiod v2 ouvre /dev/gpiochip0 en exclusivité. Si un ancien processus server.py est encore en vie, le nouveau démarrage échoue. Le script setup_autonomy.sh tue automatiquement les anciens processus avant de démarrer le service. En cas de problème manuel :

pkill -SIGTERM -f "python3.*server.py"
sleep 2
sudo systemctl restart rpi-plc

📖 Exemples de programmes

Exemple 1 — Régulation de température simple

Maintenir une température à 60°C avec hystérésis de 2°C :

// Page 1 — Lecture sonde et comparaison
SENSOR(ref=ANA0) → RF0    // Sonde 1 → RF0
COMPARE_F(RF0, seuil=60, hyst=2, op=lt) → M0   // T < 60°C ?

// Page 2 — Activation relais
COIL(EN=M0) → Q → OUTPUT(GPIO=17, "Chauffe K1")

Exemple 2 — Programmation horaire Jour/Nuit

Basculer entre deux consignes selon l'heure :

// LOCALTIME fournit HOUR → RF13
LOCALTIME → HOUR=RF13

// CArithm : activer si entre 8h et 22h
// A1=RF13 (heure), od1 = mode jour
if (A1 >= 8 && A1 < 22) od1 = 1;
else od1 = 0;

// MUX : Consigne jour (22°C) ou nuit (17°C)
MUX(IDX=M0, IN0=RF_nuit=17, IN1=RF_jour=22) → RF_consigne

Exemple 3 — Comptage horaire (RunTimeCnt)

Comptabiliser les heures de marche d'un circulateur :

// Sortie K9 (GPIO 27) = état circulateur
OUTPUT(GPIO=27) → Q → RUNTIMCNT.RUN
RUNTIMCNT → STARTS=RF10, TOTAL=RF11, RUNTIME=RF12

🔧 Calibration des sondes NTC40

Chaque sonde NTC40 peut présenter un léger écart de fabrication (±1 à ±3°C typiquement). La calibration permet de corriger cet écart individuellement, sans modifier le câblage.

Paramètres de calibration par sonde

ParamètreValeur par défautDescription
NomANA0…ANA11Nom affiché partout (SCADA, Telegram, rapports)
Offset (°C)0.0Valeur ajoutée après la mesure : T_cal = T_brut + offset
Gain (×)1.000Facteur multiplicatif : T_cal = T_brut × gain + offset
Alarme haute (°C)90.0Seuil d'alarme individuel (remplace le seuil global)
Alarme basse (°C)3.0Seuil bas individuel

Formule de correction

T_calibrée = T_brute × gain + offset

# Exemple : sonde affiche 22.5°C, valeur réelle 23.2°C
# → offset = +0.7°C
T_calibrée = 22.5 × 1.0 + 0.7 = 23.2°C

# Exemple : sonde linéairement déviante (+0.5% de déviation)
# → gain = 1.005
T_calibrée = 50.0 × 1.005 + 0.0 = 50.25°C

La calibration est appliquée automatiquement avant toute émission (SCADA, Telegram, historique SQLite, rapports). La valeur brute reste disponible dans le champ celsius_raw de l'API.

Procédure de calibration pratique

1
Mesurer avec un thermomètre de référence — placer les deux sondes dans le même bain à température stabilisée.
2
Lire la valeur brute dans l'onglet Calibration du SCADA (champ "brut").
3
Calculer l'offset : offset = T_référence − T_brute. Entrer la valeur et sauvegarder.
4
Vérifier que la valeur calibrée correspond à la référence.

Les calibrations sont stockées dans rpi_server/calibration.json et persistent aux redémarrages.

🐍 Bloc PYBLOCK — Code Python natif

Le bloc PYBLOCK permet d'écrire de la logique PLC en Python 3 natif, avec accès complet aux variables du moteur (registres RF, mémoires M, sondes ANA, GPIO). Contrairement au CArithm (syntaxe C simplifiée), PYBLOCK exécute du vrai Python avec gestion des erreurs, imports math/statistics, et un dictionnaire d'état persistant entre cycles. Idéal pour les algorithmes complexes, le filtrage, les statistiques et les logiques temporelles avancées.
💡 Double-clic sur un bloc PYBLOCK dans le canvas FBD pour ouvrir l'éditeur flottant redimensionnable avec numéros de lignes, coloration et raccourci Ctrl+S.
⚠ Sécurité : Les __builtins__ sont restreints (pas de open(), exec(), import dynamique, accès réseau). Les imports autorisés sont math, datetime, statistics — disponibles directement sans import.

Exemples commentés

Exemple 1 — Filtre passe-bas (moyenne glissante)

# Filtre passe-bas sur A1 — constante de temps tau configurable tau = 30.0 # secondes # Initialiser l'état au premier cycle if 'filtered' not in state: state['filtered'] = A1 # Formule filtre RC discret : alpha = dt / (tau + dt) alpha = dt / (tau + dt) state['filtered'] = alpha * A1 + (1 - alpha) * state['filtered'] OA1 = state['filtered'] # valeur filtrée → registre RF

Exemple 2 — Compteur horaire avec reset

# Cumule le temps de marche (d1=signal marche) en heures → OA1 # d1 = True quand l'équipement tourne # d2 = True pour reset du compteur if d2: # reset state['hours'] = 0.0 elif d1: # en marche state.setdefault('hours', 0.0) state['hours'] += dt / 3600.0 # dt en secondes → heures OA1 = state.get('hours', 0.0) write_register("RF10", OA1) # écriture directe en plus du port OA1

Exemple 3 — Loi d'eau personnalisée

# Calcul de la consigne départ selon température extérieure # A1 = T extérieure, A2 = consigne ambiante # OA1 = consigne départ calculée T_ext = A1 # température extérieure °C T_amb = A2 # consigne ambiante °C (ex: 20°C) # Loi d'eau linéaire entre deux points : # T_ext=-10 → départ=65°C, T_ext=+15 → départ=35°C T_dep_min = 35.0 # départ à T_ext haute T_dep_max = 65.0 # départ à T_ext basse T_ext_min = -10.0 T_ext_max = 15.0 ratio = (T_ext - T_ext_min) / (T_ext_max - T_ext_min) ratio = max(0.0, min(1.0, ratio)) # clamp 0..1 OA1 = T_dep_max - ratio * (T_dep_max - T_dep_min) # Correction selon écart ambiance réelle vs consigne T_amb_reel = read_analog("ANA7") correction = (T_amb - T_amb_reel) * 2.0 # 2°C départ par °C d'écart ambiance OA1 = max(25.0, min(75.0, OA1 + correction))

Exemple 4 — Détecteur de gel avec hystérésis et temporisation

# A1 = température extérieure # od1 = alarme gel avec hystérésis + délai confirmation SEUIL_GEL = 2.0 # °C — déclenchement gel SEUIL_DEGEL = 5.0 # °C — fin alarme gel DELAI_S = 120 # secondes de confirmation avant alarme if A1 < SEUIL_GEL: state.setdefault('timer_gel', 0.0) state['timer_gel'] += dt od1 = state['timer_gel'] >= DELAI_S # alarme après DELAI_S s else: if A1 > SEUIL_DEGEL: state['timer_gel'] = 0.0 # reset timer seulement si bien au-dessus od1 = False # plus d'alarme write_signal("M5", od1) # écriture directe bit mémoire M5

Exemple 5 — Statistiques températures (min/max/écart)

# A1..A4 = 4 sondes de température # OA1 = moyenne, OA2 = écart-type, od1 = alarme si écart > seuil temperatures = [A1, A2, A3, A4] valides = [t for t in temperatures if t > -50] # filtrer sondes débranchées if valides: OA1 = statistics.mean(valides) OA2 = statistics.stdev(valides) if len(valides) > 1 else 0.0 od1 = OA2 > 10.0 # alarme si écart > 10°C entre sondes else: OA1 = OA2 = 0.0 od1 = True # alarme si toutes les sondes KO
🔑 Différences clés PYBLOCK vs CArithm :
• Python natif (indentation, list comprehensions, f-strings...) vs syntaxe C simplifiée
• state persistant entre cycles vs pas de mémoire inter-cycles dans CArithm
• read_analog() / write_register() accessibles directement
• math, statistics, datetime disponibles
• Messages d'erreur détaillés dans le journal PLC
